# WillDeep Android Mobile Gateway Requirements

> Last updated: 2026-06-14 | Android version: v1.1.0-rc1 | Protocol: mobile-gateway.v1

## Summary

WillDeep Android is the mobile control surface for the Mac desktop app. The Android app connects to the Mac-side Go Mobile Gateway over the local network, displays Agent session state, and sends user commands back to the Mac. The Mac remains the execution authority for Agent turns, tool approvals, patch application, workspace access, MCP trust, and all safety gates.

This repository owns the native Android client. The current implementation uses Kotlin, Jetpack Compose, Material 3, OkHttp WebSocket, Kotlin coroutines, and encrypted local storage for the long-lived device token.

## Goals

- Pair with a Mac by consuming the short-lived QR payload generated by WillDeep Settings.
- Exchange the pairing token for a long-lived device token through `POST /mobile/pair/claim`.
- Store the device token securely on Android.
- Connect to `GET /mobile/ws` using `Authorization: Bearer <device_token>`.
- Subscribe to session snapshots, session updates, message deltas, acknowledgements, and errors.
- Send mobile commands such as session listing, session creation, message sending, and turn stop.
- Keep all core execution on the Mac. Android is a display and control client, not an Agent runtime.

## Non-Goals

- No cloud relay, public tunnel, or account sync.
- No model provider credentials on Android.
- No direct Android access to filesystem, shell, patch, or MCP APIs.
- No bypass of desktop Plan gate, danger confirmation, workspace path guard, tool approval, patch review, or MCP trust flows.

## Current Android Scope

Implemented in v1.0.0-rc1:

- Compose-first single-screen client.
- QR pairing scan through CameraX and ML Kit barcode scanning.
- Manual pairing payload paste as a fallback path.
- Encrypted token persistence through AndroidX Security.
- WebSocket connection with `X-App-Version` populated from `BuildConfig.VERSION_NAME`.
- Session list refresh, session creation, session selection, message send, and turn stop commands.
- Event log for mobile messages, Mac deltas, gateway ack, and gateway errors.
- Localized user-visible UI strings in `res/values/strings.xml`.

Planned next:

- Dedicated approval panels for `tool.pending`, `patch.upsert`, and job controls.
- Multi-language resource sets.
- Instrumented integration test against the Mac gateway mock.
- Better offline/reconnect state handling.

## Pairing Payload

The Mac-side gateway exposes a desktop-only pairing payload. Android expects the QR content to be JSON:

```json
{
  "base_url": "http://192.168.1.20:8876",
  "pairing_token": "short-lived-token",
  "protocol_version": "mobile-gateway.v1",
  "desktop_name": "Rocky's Mac",
  "expires_at": "2026-06-13T12:02:00Z"
}
```

The token is short-lived and single-use. Android must not log the full payload or device token.

## HTTP API

Android currently calls:

| Method | Path | Purpose |
| --- | --- | --- |
| `POST` | `/mobile/pair/claim` | Exchange the short-lived pairing token for a device token. |
| `GET` | `/mobile/ws` | Establish the authenticated WebSocket event and command channel. |

The client sends `X-App-Version` on gateway HTTP and WebSocket requests. The value must match `app/build.gradle.kts` `versionName`.

## WebSocket Envelope

All messages follow the shared envelope:

```json
{
  "id": "message-id",
  "type": "message.send",
  "session_id": "session-id",
  "payload": {},
  "ts": "2026-06-13T12:00:00Z"
}
```

Android sends:

- `session.list`
- `session.create`
- `session.select`
- `message.send`
- `turn.stop`

Gateway events parsed by Android v1.0.0-rc1:

- `state.snapshot`
- `session.upsert`
- `message.delta`
- `ack`
- `error`
- `command.error`

Unknown events are ignored for now so the Mac can add event types without breaking this client.

## Security Requirements

- Device token must be stored with encrypted preferences.
- Bearer token and pairing payload must not be printed to logs.
- LAN cleartext traffic is allowed only because the Mac gateway is local-network HTTP during the current phase.
- Android must treat every command as a request to the Mac; it must never execute local workspace or shell actions itself.
- Device revocation is enforced by the Mac gateway. When the gateway closes or rejects the WebSocket, Android should return to a disconnected or error state.

## Acceptance Criteria

- The app builds with `./gradlew :app:assembleDebug`.
- The first screen is the usable Compose gateway client, not a marketing page.
- User-visible text is stored in Android string resources.
- Pairing payload claim stores a long-lived device token securely.
- A paired device can open `/mobile/ws`, send `session.list`, and display returned session state.
- Version `1.0.0-rc1` is visible in Gradle metadata and sent through gateway request headers.
